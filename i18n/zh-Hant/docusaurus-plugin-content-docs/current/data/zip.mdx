### ZIP

ZIP 是一種支援無損資料壓縮的檔案格式。ZIP 檔案可以包含一個或多個檔案或目錄，這些檔案或目錄可以使用 ZIP 規範中提供的多種壓縮演算法進行壓縮。最常用的壓縮演算法是 DEFLATE，這也用於 [gzip](../data/gzip.mdx) 和 [PNG](../images/PNG.mdx)。

DEFLATE 透過 LZ77 無損編碼和 Huffman 編碼的結合來進行壓縮。首先，DEFLATE 使用 LZ77 來尋找資料中的模式並減少冗餘，接著使用 Huffman 編碼對由 LZ77 發現的模式進行編碼，將較常見的模式分配較小的位元值。

此外，ZIP 檔案中的每個檔案都是獨立壓縮的，因此可以在不對整個檔案進行壓縮或解壓的情況下提取現有檔案或新增檔案。

ZIP 的兼容性幾乎是無所不包的。傳統 ZIP（ZIP 規範中的壓縮方法 8）將壓縮檔案的大小限制為 4 GB，不過大多數 ZIP 壓縮工具使用 Deflate64(tm)（ZIP 規範中的壓縮等級 9）來繞過此限制。ZIP 與 [gzip](../data/gzip.mdx) 具有競爭力，並且已被 [bzip2](../data/bzip2.mdx)、[XZ](../data/xz.mdx)、[7-zip](../data/7z.mdx)、[brotli](../data/brotli.mdx)（在某種程度上）和 [Zstandard](../data/zstd.mdx) 等格式和演算法多次取代。

## 格式分解

DEFLATE 是基於 LZ77 的壓縮器，它會在輸入資料中尋找重複的字節序列，並用對先前出現的短引用來替換這些序列。它還使用 Huffman 編碼對符號進行變長編碼，根據符號出現的頻率進行編碼。DEFLATE 對於每個壓縮數據塊有兩種模式：這些模式被指定為「靜態」或「動態」Huffman 壓縮塊。在靜態模式下，Huffman 編碼是固定且預定義的。在動態模式下，Huffman 編碼會動態生成並與壓縮數據一起傳輸。

ZIP 檔案具有特定的結構，包括四個主要的檔案標頭組件：本地檔案標頭、中央目錄檔案標頭、中央目錄結束記錄和數據描述符。本地檔案標頭儲存有關每個壓縮檔案的信息，如其名稱、大小、CRC-32 檢查碼、壓縮方法和可選的額外欄位。中央目錄檔案標頭儲存類似於本地檔案標頭的信息，但還包括 ZIP 檔案中每個本地檔案標頭的偏移量。中央目錄結束記錄標記 ZIP 檔案的結束，並包含有關中央目錄檔案標頭的數量和大小的信息。數據描述符是可選的，儲存有關壓縮數據的額外信息，如 CRC-32 檢查碼、未壓縮大小和壓縮大小。

ZIP 檔案也可以支援其他壓縮方法，如 Deflate64(tm)、BZIP2、LZMA 和 [Zstandard](../data/zstd.mdx)。這些方法在大多數 ZIP 工具中不廣泛支援，可能會導致兼容性問題。ZIP 檔案也可以包含未壓縮的數據。該格式還支援加密以保護數據免受未授權訪問。ZIP 支援兩種類型的加密：傳統的 ZipCrypto 加密和強加密。ZipCrypto 加密被認為是不安全的，而 ZIP 中的強加密使用更強健的演算法，但分佈在多個標準中。因此，強加密並未標準化，可能會導致兼容性問題。

ZIP 檔案可以通過更高效地表示冗餘數據來減少檔案和資料夾的大小。它們還可以將多個檔案和資料夾組合成一個單一的檔案，便於轉移或存儲。沒有比 ZIP 更受歡迎的實現來進行這種一般資料壓縮。ZIP 檔案還可以保留原始檔案的元數據，如名稱、路徑、日期和屬性。然而，ZIP 檔案也有一些限制和問題。例如，傳統 ZIP 檔案（即非 ZIP64）對每個壓縮檔案有 4 GB 的大小限制和每個檔案 65,535 個條目。大多數 ZIP 實現不支援檔案內的符號連結或硬連結。

此外，ZIP 可以以多種不同的方式進行編碼。Apple 在 Finder 中提供了預設的「壓縮」選項，可以將選定的檔案壓縮成 ZIP 檔案，許多 Linux 桌面環境也提供了易於創建 ZIP 檔案的 GUI 功能。在 Windows 上，常見的壓縮方式是使用 7-zip 數據壓縮和解壓縮工具（與 [7-zip 壓縮格式](../data/7z.mdx) 相關，但兩者不同）。

## 歷史

ZIP 格式由 Phil Katz 開發，作為一種開放格式和開放規範，他的實現 PKZIP 是共享軟體。

[限制版 ZIP 格式](http://www.digitalpreservation.gov/formats/fdd/fdd000361.shtml) 存在，並用於其他檔案類型，如 Java .jar 檔案、許多 Microsoft Office 檔案格式、Office Document Format 檔案（.odt、.ods、.odp）和電子書閱讀器的 EPUB 檔案。

大約在 1990 年，Info-ZIP 出現了。「Info-ZIP 的目的是提供免費、可攜式、高品質的 Zip 和 UnZip 壓縮-解壓縮工具，與基於 DOS 的 PKZIP 兼容」（https://infozip.sourceforge.net/）。他們成功地實現了這一目標，導致 ZIP 格式的採用增加。

在 1990 年代初期，[gzip](../data/gzip.mdx) 格式被開發出來，它源於 Info-ZIP 工具中的 Deflate 代碼。它旨在取代 Unix 的 `compress` 工具，該工具使用當時已獲專利的 LZW 壓縮算法，這威脅到其免費使用。雖然 Deflate 的一些具體實現已由 Phil Katz 獲得專利，但該格式本身並未獲得專利，因此編寫了一個不侵犯專利的 Deflate 實現。

與 `.tar` 不同，`.zip` 具有位於檔案末尾的中央目錄，提供內容清單。這和單獨壓縮的特性提供了對 `.zip` 檔案中單獨條目的隨機訪問。而 `.tar` 檔案需要解壓並從頭到尾掃描才能建立目錄。

流行的 `tar` 工具可以創建檔案存檔，並提供直接壓縮到 `.tar.gz` 格式的選項，這是 gzip 的一個非常受歡迎的使用場景。由於 `.tar` 的壓縮可以利用跨檔案的冗餘，ZIP 通常壓縮效果不如 tar 和 gz 的組合。`.tar.gz` 是 Unix 上最常用的存檔格式，由於其非常高的可攜性，但也有更好的壓縮方法可用。一些包括 [XZ](../data/xz.mdx)、[bzip2](../data/bzip2.mdx)、[brotli](../data/brotli.mdx)、[7-zip](../data/7z.mdx) 和 [Zstandard](../data/zstd.mdx)。

在這種情況下，ZIP 的優勢在於它會單獨壓縮檔案並在檔案末尾構建中央目錄，提供對 `.zip` 檔案中單獨條目的隨機訪問。相比之下，`.tar` 檔案必須解壓並從頭到尾掃描才能建立目錄。

## Encoding

#### Linux & macOS

To encode to a ZIP file most efficiently on Linux or macOS, it is worth using the 7-zip implementation of DEFLATE for ZIP compression. The 7zip website's homepage claims: "For ZIP and GZIP formats, 7-Zip provides a compression ratio that is 2-10 % better than the ratio provided by PKZip and WinZip." You can use the highly flexible 7-zip CLI utility through binaries available on [7-zip's Download page](https://www.7-zip.org/download.html). Here are some direct download links: [Linux x86_64](https://www.7-zip.org/a/7z2301-linux-x64.tar.xz) | [macOS Universal](https://www.7-zip.org/a/7z2301-mac.tar.xz)

Once you've downloaded the utility, remember whether you are choosing to use the `7zz` binary or the static `7zzs` binary. Commands run using `7zz` should run using `7zzs` as well, so please replace `7zz` in our examples as appropriate if you choose not to use it. Additionally, please copy your choice of binary to your `/usr/local/bin` if you want to be able to use it everywhere.

- To encode a ZIP file at the lowest effort setting using one thread:
```bash
7zz a -bso0 -tzip -mmt1 -mx1 "Output.zip" "Input"
```

- To encode a ZIP file at the highest effort setting using eight threads:
```bash
7zz a -bso0 -tzip -mmt8 -mx9 "Output.zip" "Input"
```

## Conclusion

The only real benefit of using ZIP over more modern formats currently is compatibility. It may be viable when compared to 7z & XZ due to a reduction in complexity that improves encode & decode speed, but Zstandard is incredibly performant in both of these areas and generally outperforms ZIP. When it comes to content delivery on the Web, Brotli has been adopted across all modern web browsers and offers a better alternative to older compression technologies used on the Web that resemble ZIP.
